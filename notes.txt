        -> A configuration management database (CMDB) is a centralized file that 
        functions as a comprehensive data warehouse, organizing information about an IT environment. 
        CMDB clarifies the relationships between hardware, software components, 
        and networks for improved configuration management.

        -> https://www.servicenow.com/in/products/it-operations-management/what-is-cmdb.html

        -> "CMDB" is Heart Of Service Now ITOM 

        -> CMDB is ServiceNow's Master Inventory of everything in your IT environment

            all of these lives as records 

    
    -> Configuration Items (CI)

        -> Configuration Items are "atoms of CMB"

        -> A Configuration Item (CI) in ServiceNow is a digital record within the 
        Configuration Management Database (CMDB) representing "any manageable IT component", 
        such as servers, applications, services, or network devices.

        -> CI's Track Infrastructure Details (Manufacturer, Vendor, Locations) 
        to support (ITIL Process)

        -> ITIL

            -> Information Technology "Infrastructure Library"

            -> ITIL processes are structured sets of activities within the ITIL framework that guide 
            IT service management (ITSM) to align IT with business needs. 
            They cover the full service lifecycle, including planning, delivery, operation, and improvement. 

            -> NO ITIL 

                -> Incidents -> Handled Randomly

            -> ITIL 

                -> Structured Approach

        Controls changes in system

        4. Asset Management                     ->      ITAM Module

    -> Credentials 

        -> Once the MID Server reaches your device on the network, it needs to
        log into the servers to interrogate it, that's what Credentials are.

        -> Credentials are authentication secrets stored securely in service now 
        that the MID Server uses to access devices. 

        -> Credentials are stored in "discovery_credentials" table 

        -> Credentials are never sent in plain text, they are "encrypted" 
        using service now encryption.

        -> MID Server receives encrypted Credentials and decrypts it locally at
        the moment of use 

        -> There are several types of Credentials 

            -> SSH Credentials used in Linux & Unix environments
            SSH Credentials are made up of: Username, Public Key & Private Key.
            Password less authentication(way secure then Password based) uses
            Public Key & Private Key for accessing Linux servers.

            -> Widows Credentials uses Username & Password 

            -> JDBC Credentials (Database Connectivity)

            -> VM Ware (VM Ware Credentials)

            -> Cloud Infrastructure Like AWS, AZURE & GCP, uses API Keys or Tokens  



    -> Probes & Sensors 

        -> Probes & Sensors are two halves of same job 

    -> Probes 

        -> A Probe is a Question

            -> It's Script / Command that MID Server executes against target device 
            to collect RAW Data 

            -> It runs on MID Server Side, reaches to target device and brings 
            back all the raw output like commands, XML, JSON, WMI Queries etc 

            -> The raw output will be stored at service now side inside 
            ECC(External Communication Channel) Queue.


    -> Sensors 

        -> A Sensor is an Answer 

            -> It's Script(javascript) that takes raw output from Probe 
            and parses it, extracting fields like OS, IP, Hostname, Port etc 
            and mapping them to CI Attributes, updating as records in CMDB.


    -> Detective Story 

        -> Detective Assistant / Probe 

            -> Goes to the crime scene, takes photos, lifts fingerprints etc
            collected all raw evidence 

            -> ITOM: Probe runs scripts, collects data 

        -> Bag Of evidence / ECC Queues 
            
            -> Bag Of evidence contains full od unprocessed raw evidence items 

            -> ITOM: RAW Data collected i.e unprocessed data 

        -> Detective / Sensor 

            -> Detective sits at desk, opens the Bag Of evidence, 
            examines each item, extracts meaning of those items 
            and writes final case report 

            -> ITOM: Extracts Data from ECC Queue and gives meaning fields
            like Hostname, OS, IP etc 


            ServiceNow says → "Go discover this IP"
                    ↓
            MID Server sends PROBE → runs commands/scripts on target device
                    ↓
            Raw data drops into ECC QUEUE (ecc_queue table)
                    ↓
            SENSOR picks it up → reads, parses, extracts
                    ↓
            CMDB gets updated with new/updated CI records


    -> Patterns 

        -> Patterns are servicenow's modern replacement for older probes/sensors 
        approach for application and software services discovert 

        -> Probes & Sensor are still heavily used for infrastructure (Hardware/OS/NW)

        -> NOTE: Patterns are preferred method for discovering Application and Services

        -> A Pattern is a Declarative Definition, unlike Imperative Definition of
        Probes & Sensors 

        -> Patterns discover Specific Type of software or service

            -> Defines: What to look for, which command to run, what files to read,
            how to map the results to CI Attributes 

            -> Imperative Definition means "How to do"

            -> Declarative Definition means "What to do"

        -> Pattern are written in Visual Designer in servicenow(Pattern Designer)
        rather than raw javascript

    -> CI Classification 

        -> CI Classification is the final step, the "Classifier Engine" or "Decision Engine"
        determines What CI Class to create and Whether To Create a new CI or Update an existing Once

        -> When a Sensor or Pattern finished parsing data from device, it hands over 
        raw Attributes to CI Classifier Engine

        -> The Classifier's job is to look at those Attributes and Answer two questions 

            -> What Type Of CI is this ?

            -> Does it already exist in CMDB ?

        -> NOTE: Classifier Engine == Identification Rules 

        -> Identification / Reconciliation 

            -> Identification rules for matching Configuration Items before insert/update

    
    -> IRE (Identification and Reconciliation Engine )

        -> Identification and Reconciliation Engine is underlying servicenow Engine
        that uses Identifier Rules 

            -> Each Class has a defined set of "Identifier Fields"  that together makes
            a CI Unique 

            -> "Criterion Attributes" are like Unique fingerprints of a CI 

                -> For a web server like Apache, the combination of 
                sys_class_name + config_file + version  is the identifier 

                -> For a Docker Container, the container_id  is the identifier 

                -> For Java Server like Tomcat, the combination of 
                sys_class_name + install_directory + running process

        -> NOTE: If IRE finds a match, it updates the existing CI this is called 
        Reconciliation. If no match is found, creates a new CI.

        -> IRE - Questions it always asks  ?

            -> Question 1: What are you ?

                -> Linux Server / Windows Linux / Apache Web Server / Container / Tomcat Server ??

            -> Question 2: Have i seen you BEFORE ? 

                -> Does this CI already exist in CMDB 

                        YES -> UPDATE existing CI 

                        NO -> CREATE new CI 

        -> Complete Flow (***)

            -> Discovery Scan To CMDB Update 

            -> Discovery Scans Server 

                -> Patterns / Sensors collects RAW Attributes
                    (config_file + version)

                -> IRE receives the payload 

                -> Question 1: What TYPE of CI ?

                    -> Looks at CI Identifiers list 
                    -> Matches "Apache" rules
                    -> Class = cmdb_ci_apache_web_server
                
                -> Question 1: Does it exist ?

                    -> Sues Criterion Attributes 

                    -> Searches cmdb_ci_apache_web_server

                    Found ? -> Update Existing CI 

                    Not Found ? -> Create New CI 

            -> CMDB is Updated

        -> Service mapping is used to discover and track IT applications and components

        how their underlying IT applications and related components connect to business services.

        -> NOTE: Service mapping is a way of more clearly understanding "application services". 

    -> Application Service 

        -> One example of an application service is financial reporting. 
        When an organization needs to generate a financial report, 
        they employ an application service that includes computer hardware, web server, 
        application server, databases, middleware, and network infrastructure components. 
        Each of these works together to provide the service of creating the desired financial report.

        -> Traditionally, IT departments have relied on horizontal discovery to 
        map IT infrastructures. 

        -> Traditionally, IT departments have relied on horizontal discovery to map IT infrastructures.
        This delivery method creates inventories that act as though 
        every application and device is standalone, without making note of the 
        connections between devices and application services. 
        
        -> Service mapping forgoes this outdated approach, 
        relying instead on top-down mapping that focuses on the 
        connections between devices and applications so that 
        any issues or changes can be immediately identified along with their impact 
        on the rest of the application service.

    -> Event Management 

        -> ServiceNow Event Management is an application that helps you keep your 
        IT Systems healthy by spotting problems 

        -> Event Management collects events from different sources, 
        figures out what is causing the issues and
        converts these issues into Alerts. 

        -> Event Management seamlessly integrates with various "Monitoring Tools"
        via Connectors, enabling comprehensive data collection 

        -> NOTE: Event Management is the process of collecting and analyzing 
        events generated by IT Systems 

    
    -> Events 

        -> An event is any change in the state of a system 

            -> CPU Usage goes beyond 90% 
            -> Disk space is full 
            -> Application service not responding 
            -> etc 

        -> Say we have vast IT Infrastructure with 

            -> 5000 Linux Servers 

            -> 500 Database Servers 

            -> 2000 Web Applications 

            -> Switches, Routers, Gateways, Firewalls etc 

        -> Your IT environment generates thousands of alerts everyday 

            -> CPU high on Server 1023
            -> CPU high on Server 2022
            -> Memory(RAM) is low on Server 95 
            -> Apache Server not responding on Servers - (1000-1500)
            -> Network Packet loss (2000-2500)
            -> 100 other scenarios 
        
        -> NOTE: Take a simple scenario of one server 

            -> 23:01:01 Server A CPU is at 90% - Coming From Prometheus Monitoring Tool 
            -> 23:01:05 Server A CPU is at 91% - Coming From Prometheus Monitoring Tool 
            -> 23:01:10 Server A CPU is at 92% - Coming From Prometheus Monitoring Tool 
            -> 23:01:15 Server A CPU is at 93% - Coming From Prometheus Monitoring Tool 
            .
            .
            .
            -> 23:02:05 Server A CPU is at 98% - Coming From Prometheus Monitoring Tool 

        -> Without Event Management, your teams get flooded with 1000's of alerts 
        for what is really just 1 problem 
        (A Server With HIGH CPU that's causing downstream issue)

        -> Event Management Job is 

            -> Take 100's of Noisy Alerts and give 1 meaningful Incident

        -> Without Event Management 

            -> Operations Team Drowns in alerts 

            -> Critical Issues get missed 

            -> Wring Team get notified 

        -> Event Management  

            -> Noise is filtered 

            -> Right Teams Get Notified 

            -> Incidents Auto Created 

    
    
    -> Alerts vs Events 

        -> Event is a raw signal coming from any monitoring tool 

            -> Your Phone Battery is at 10% then you get notification

                -> This notification is an EVENT 

            -> Prometheus Tool Sends 

                {
                    source: prometheus,
                    node: ip-address-server,
                    type: CPU,
                    message: CPU Utilization Above 90
                }

                -> NOTE: Once integration is done, above event lands in 
                em_event table in ServiceNow 


        -> Alert is something that needs your attention 

            -> An alert is deduplicated, enriched version of events 

        -> NOTE: Key Difference 

            -> EVENT - every single signal from monitoring tool 
                (could be 100's for same problem)

            -> ALERT - one consolidated record representing one ongoing problem
                (1 Alert for all those 100 CPU events)

            -> Event = Every Rain drop falling 

            -> Alert = Weather warning issued 

        
        -> Alert Levels 

            -> 1 means Critical (production down)
            -> 2 means Major (significant impact)
            -> 3 means Minor (degraded performance)
            -> 4 means Warning (keep watch on this)
            -> 5 means Clear (problem resolved)

    -> Create Alert Rules File 

        -> sudo vi /etc/prometheus/alert_rules.yml

            groups:
            - name: apache_alerts
                rules:
                - alert: ApacheDown
                    expr: apache_up == 0
                    for: 10s
                    labels:
                    severity: critical
                    annotations:
                    summary: "Apache is DOWN on {{ $labels.instance }}"
                    description: "Apache web server is not responding"

                - alert: ApacheHighConnections
                    expr: apache_accesses_total > 100
                    for: 1m
                    labels:
                    severity: warning
                    annotations:
                    summary: "Apache high connections on {{ $labels.instance }}"


    -> Above file contains conditions that tell prometheus when to panic and send alerts

        name: apache_alerts
        alert: ApacheDown (Alert Name)
        expr: apache_up == 0 (1 means apache Is running & 0 means apache is down)
        if apache_up becomes 0, then raise alert 
        severity: critical (alert importance level)
            -> critical - Very Serious
            -> warning - Medium Issue 
            -> info - Just Information
        annotations: Display Friendly Alert message
        $labels.instance - Dynamic Server name 



        alert: ApacheHighConnections (alert for too many connections)
        expr: apache_accesses_total > 100 (if apache receives more than 100 connections)
        for: 1m(check for 1 min, avoiding temporary spikes)
        severity: warning (website is still working - but system might overload soon)

        -> NOTE: Rules file contains

            -> expr         -       What Problem ?
            -> for          -       How long to wait ?
            -> severity     -       How serious ?
            -> summary      -       What message to display ?

                ServiceNow ITOM receives Event 

                    Event -> Alert 

                        Alert -> Incident
